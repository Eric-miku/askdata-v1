# AskData 协作、评审与发布流程

本文件定义后续团队协作流程。当前工作区的本地修改尚未形成 PR 或评审记录，因此不能用本文档替代真实 Git/平台证据。

## 分支策略

- `main`：始终保持可部署，只通过已评审 PR 更新。
- `feature/<issue>-<topic>`：功能和文档变更。
- `fix/<issue>-<topic>`：缺陷修复。
- `release/<version>`：只在需要稳定窗口时创建，禁止混入新功能。
- `hotfix/<issue>`：从生产 tag 创建，合并后同步回 `main`。

分支应短生命周期，定期合并 `main`，避免长期分叉和大规模冲突。

## 提交规范

每个提交保持单一目的，使用祈使式主题，例如：

```text
feat(data-source): persist schema fingerprints
fix(security): enforce field policy during replay
test(export): cover formula injection
docs(ops): document state recovery
```

禁止提交 `.env`、真实凭据、数据库快照、完整评测报告、构建产物和本地状态文件。

## PR 必填内容

1. 关联需求编号或问题链接。
2. 用户可观察的行为变化和非目标。
3. 数据模型/API/配置兼容性说明。
4. 安全与隐私影响，尤其是身份、权限、SQL、导出和日志。
5. 已执行的精确测试命令及结果。
6. UI 变更的桌面和移动端截图或浏览器验证记录。
7. 迁移、部署、回滚步骤和外部依赖。

PR 作者不得作为唯一批准人。涉及 SQL 安全、权限、凭据、导出或生产迁移时，至少需要一名对应领域评审人。

## 评审检查表

- 行为是否对应计划书和验收矩阵，而不是只满足现有测试。
- 用户输入、模型输出和 SQL 是否在信任边界处验证。
- 查询、回放、导出、推荐和执行计划是否共享当前权限。
- 数据库路径、凭据、SQL 参数和结果明文是否可能进入响应或日志。
- 状态表迁移是否兼容旧版本，失败是否可恢复。
- 前端加载、空、错误、移动端和长文本是否可用。
- 测试是否覆盖回归面，文档是否能由新成员执行。

## 发布门禁

候选版本必须满足：

```bash
uv run pytest -q
cd frontend && npm test -- --run
cd frontend && npx tsc --noEmit
cd frontend && npm run build
docker compose config
docker compose up --build
```

随后执行机器可读核心验收、危险 SQL、权限绕过、导出、Schema 变更、多用户隔离、外部模型失败和真实业务题集。阻断级/严重级缺陷必须为 0。

Docker、真实数据库、SSO 和业务准确率门禁必须在具备相应外部环境的流水线或验收环境中执行；本地单元测试不能替代这些证据。

## 版本与发布记录

采用语义化版本。每个发布 tag 应附：

- Git commit SHA、构建时间和镜像 digest。
- 配置/状态表迁移说明。
- 自动化测试、BIRD 和业务题集摘要。
- 已知限制与风险接受人。
- 部署人、验证人、批准人和时间。
- 回滚目标版本与操作步骤。

## 部署与回滚

部署前备份 `ASKDATA_STATE_DIR`，记录当前镜像 digest 和配置。部署后检查 `/health`、`/ready`、`/metrics`，执行一个授权查询、一次禁止查询和一次导出。

回滚条件包括启动失败、就绪持续失败、权限绕过、数据泄露、查询错误率显著升高或核心业务题集跌破门槛。回滚时：

1. 停止接收新请求。
2. 恢复上一镜像 digest 和兼容配置。
3. 若状态迁移不向后兼容，按发布记录恢复状态目录备份。
4. 重新执行健康、权限和核心验收场景。
5. 保存 request ID、日志时间窗和根因分析，不删除失败证据。

AskData 不负责回滚业务数据库，因为运行账号和产品路径不应修改业务数据库。
